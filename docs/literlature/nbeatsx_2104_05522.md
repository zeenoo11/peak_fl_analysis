## Why this note

`MinimalNBEATSx` reads as a *minimal / cut-down* variant from the name. The
question that triggered this note: "is NBEATSx supposed to stack many
blocks per stack? Did we drop most of them?" The answer is no ‚Äî the
single-block-per-stack layout we use is the published NBEATSx default, not
a simplification. What we *did* trim is hidden width and the exogenous
stack. Both are intentional and called out in CLAUDE.md `## Hard-coded
shapes`.

## Architecture (paper ¬ß3 + author code)

Three-level hierarchy: `block` ‚äÇ `stack` ‚äÇ `model`.
- **Block**: a 4-FC-layer MLP ‚Üí linear projection to `Œ∏` ‚Üí basis
  function (trend / seasonality / generic / exogenous) ‚Üí `(backcast,
  forecast)`. The inputs are doubly-residual: each block subtracts its
  backcast from the running residual before passing it on.
- **Stack**: a sequence of blocks of one basis type, summing their
  forecasts. NBEATSx allows: `trend`, `seasonality`, `identity` (=
  generic), `exogenous_wavenet`, `exogenous_tcn`.
- **Model**: a sequence of stacks. Final forecast = sum across all stack
  forecasts.

## Default hyperparameters ‚Äî author's hyperopt search space

From `src/hyperopt_nbeatsx.py` `get_experiment_space()` (cchallu/nbeatsx
@ main):

```python
'n_blocks':    hp.choice('n_blocks',    [ [1, 1] ])
'n_layers':    hp.choice('n_layers',    [ [2, 2] ])
'stack_types': hp.choice('stack_types', [
                  ['identity'],
                  1*['identity'] + ['exogenous_wavenet'],
                  1*['identity'] + ['exogenous_tcn'],
                  ...
               ])
'n_hidden_1':  hp.quniform('n_hidden_1', 50, 500, 1)
'n_hidden_2':  hp.quniform('n_hidden_2', 50, 500, 1)
```

So **the *entire* hyperopt sweep fixes `n_blocks=[1,1]`** ‚Äî one block per
stack ‚Äî and only varies stack composition + width + MLP depth (n_layers=2
fixed). The interpretable variant adds a third stack: `n_blocks=[1,1,1]`,
`stack_types=['trend','seasonality', <exogenous>]`.

## Default hyperparameters ‚Äî Nixtla `neuralforecast.NBEATSx`

```
n_blocks    = [1, 1, 1]
stack_types = ['identity', 'trend', 'seasonality']
mlp_units   = 3 * [[512, 512]]      # one per stack, 2-layer MLP, hidden 512
```

Same structure: 3 stacks √ó 1 block, 2-layer MLP, hidden 512.

## Discussion: "30 blocks" confusion

`cchallu/nbeatsx#4` ("Number of stacks confusion") flags that the
*original N-BEATS* paper (Oreshkin 2020, Table 18) lists the generic
variant as "30 stacks √ó 1 block", and the issue argues this is more
naturally read as "1 stack √ó 30 blocks". NBEATSx-G in the paper is much
smaller than that ‚Äî only 2 stacks (1 generic + 1 exogenous). NBEATSx
deliberately *moves away* from the "many blocks" regime. Our
single-block-per-stack layout is on the published side of this
contraction, not a custom simplification.

## Mapping to `src/models/nbeatsx.py:MinimalNBEATSx`

| NBEATSx (Olivares / Nixtla default)               | `MinimalNBEATSx`                          | Match? |
|---------------------------------------------------|-------------------------------------------|---|
| 3 stacks: identity / trend / seasonality          | 3 stacks: trend / seasonal / generic      | ‚úì same set, generic ‚Üî identity |
| `n_blocks = [1, 1, 1]`                            | one `NBEATSxStack` instance per stack      | ‚úì |
| MLP depth 2 (Olivares hyperopt) / variable        | MLP depth 4 (`fc1..fc4`) per block         | ‚úó deeper than Olivares default; matches original N-BEATS' 4-layer block |
| Hidden 50‚Äì500 (paper) / 512 (Nixtla)              | `D_MODEL = 64`                             | ‚úó much narrower |
| `n_polynomials` (trend basis order): paper 3      | `N_POLYNOMIALS = 3`                        | ‚úì |
| `n_harmonics` (seasonality basis): paper 1        | `N_HARMONICS = 5`                          | ‚úó wider seasonal basis |
| Exogenous stack (`exogenous_wavenet/tcn`)         | none                                       | ‚úó univariate input only |
| Doubly-residual stacking                          | yes (`residual = residual - backcast`)     | ‚úì |
| `forecast = Œ£ stack forecasts`                    | yes                                        | ‚úì |

The two intentional deviations from Olivares default that matter for the
project:
1. **Hidden = 64** instead of 512. Frozen because the post-hoc Peak-VQ
   codebook (`src/models/vq_kmeans.py`, M=32, D=64) operates on
   `h_generic` ‚àà ‚Ñù‚Å∂‚Å¥. Widening backbone would force a codebook
   re-fit and break v10 b2 checkpoint compatibility.
2. **No exogenous stack**. UMass Smart\* 2016 inputs in our pipeline are
   univariate hourly load ‚Äî no weather / calendar tensors are joined in
   `src/dataloader/`. NBEATSx without exogenous reduces to N-BEATS-G+I
   essentially.

What we did *not* change: number of stacks, blocks per stack, basis
types, doubly-residual stacking, summation rule. So the depth axis is
faithful to NBEATSx; only the width and exogenous axes are pruned.

## v06 implication

`NBEATSxAux` wraps the same `MinimalNBEATSx` and adds a 2-layer
`PeakAuxHead`. Setting `--aux_lambda 0` zeroes the gradient on the aux
head, so only the NBEATSx body is trained. The body is the published
NBEATSx default layout (3 √ó 1 block, doubly-residual), just narrower ‚Äî
not a different model.
