# N-BEATS — Neural basis expansion analysis for interpretable time series forecasting

arxiv: 1905.10437 (Oreshkin, Carpov, Chapados, Bengio; ICLR 2020)
Code (official): https://github.com/ServiceNow/N-BEATS
Project usage: indirect — N-BEATS is the predecessor of NBEATSx
(arxiv:2104.05522), which is the actual backbone used in v01–v05 + v06.
Kept here as the cross-reference for the "how many blocks per stack?"
question.

---

## Why this note

The intuition "NBEATS-style models stack many blocks per stack" is true
*for original N-BEATS*. NBEATSx (the variant we actually use) walks this
back to one-block-per-stack as the published default — see
`nbeatsx_2104_05522.md` for that. This note records what the *original*
N-BEATS paper specified, so the contrast is on file.

## Architecture (paper §3.3)

Two published variants:
- **N-BEATS Generic** (`G`): a single stack of `nb_blocks_per_stack`
  blocks where the basis is the identity / fully-learned projection.
  Paper Table 18: 30 blocks. Issue cchallu/nbeatsx#4 notes the paper's
  table is ambiguously phrased as "30 stacks × 1 block"; the operational
  reading is "1 stack × 30 blocks" because the model is single-stack.
- **N-BEATS Interpretable** (`I`): two stacks — a `trend` stack
  (polynomial basis) followed by a `seasonality` stack (Fourier basis).
  Paper Table 18: **3 blocks per stack** (i.e. 2 × 3 = 6 blocks total).
  Hidden width 256, with weight sharing within a stack.

In both, every block is a 4-layer fully-connected MLP feeding a linear
projection to θ → basis function → `(backcast, forecast)`. Doubly-
residual stacking is applied across blocks within a stack and across
stacks.

## ServiceNow/N-BEATS reference repository

The author release (`ServiceNow/N-BEATS`) keeps configurations in `.gin`
files under `experiments/<dataset>/`. The reusable model code lives in
`models/nbeats.py`. The constructor takes a `blocks: t.nn.ModuleList`
directly, so stack count / blocks-per-stack are decided by the
experiment driver (M3, M4, Tourism, electricity, etc.) rather than as
defaults on the class itself.

## Mapping to `src/models/nbeatsx.py:MinimalNBEATSx`

| Original N-BEATS Interpretable (Oreshkin 2020) | `MinimalNBEATSx`                | Match? |
|------------------------------------------------|----------------------------------|---|
| 2 stacks: trend / seasonality                  | 3 stacks: trend / seasonal / generic | ✗ N-BEATS-I has no generic; we add it (matches NBEATSx-I+G) |
| 3 blocks per stack                             | 1 block per stack                 | ✗ smaller than N-BEATS-I; matches NBEATSx default |
| 4-layer MLP per block                          | 4-layer MLP per block             | ✓ (NBEATSx default would be 2) |
| Hidden width 256                               | `D_MODEL = 64`                    | ✗ narrower (matches Peak-VQ codebook width) |
| Weight sharing within a stack                  | none (each stack has 1 block)     | n/a |
| Polynomial trend, Fourier seasonality bases    | same                              | ✓ |
| Doubly-residual stacking                       | yes                               | ✓ |

So `MinimalNBEATSx` is closer in shape to NBEATSx-I+G default than to
N-BEATS-I. The "many blocks per stack" feature of N-BEATS does not carry
over to NBEATSx, and accordingly does not carry over to our port.
