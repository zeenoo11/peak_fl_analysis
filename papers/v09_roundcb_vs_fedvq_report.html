v09 코드북 두 갈래 — RoundCB vs FedVQ

NBEATSx residual 흐름 × FedProto prototype memory 관점에서 본 한눈 비교 · 2026-05-26
한 문장 요약 — RoundCB는 backbone이 설명하지 못한 잔차(residual)를 사후에 사전(dictionary)으로 저장하는 post-hoc 메모리이고, FedVQ는 그 사전을 학습 중에 backbone과 함께 키우는 co-train 메모리다. 둘 다 FedProto의 count-weighted prototype 집계를 계보로 삼지만, FedVQ만이 codebook을 backbone 표현(hg)에 되먹임한다.

0. 개념적 다리 — Residual을 버리지 않고 Prototype Memory에 축적

NBEATSx는 iterative residual decomposition이다. 각 stack이 설명한 backcast를 빼고, 남은 잔차 ybacks,b = ybacks,b−1 − ŷbacks,b를 다음 stack으로 넘긴다. 잔차에는 rare pattern · client-specific dynamics · regime shift 같은 "아직 압축되지 않은 latent knowledge"가 남는다.

두 driver 모두 이 잔차를 generic stack의 hidden h_generic ∈ ℝ⁶⁴로 포착하고 (NBEATS에서 generic stack은 trend·seasonal이 못 설명한 부분을 담당 → 잔차 표현에 가장 가깝다), 이를 codebook이라는 latent dictionary에 모은다. 그리고 cluster-mean forecast 잔차를 offsets ∈ ℝ^{M×24}로 저장해 예측을 보정한다 — 보내주신 "residual → prototype memory" 그림이 코드에서 실제로 구현된 지점이다. 차이는 그 dictionary를 "언제, gradient와 함께 키우느냐"에 있다.

V9-RoundCB — 사후 잔차 사전

[FL backbone] FedAvg/Prox/Rep/Ditto/Proto NBEATSxAux, normal training │ │ ── 매 라운드 끝 (frozen forward) ── ▼ h_g 추출 → Stage-1 client KMeans(K=2) │ (centroids,counts) upload ▼ Stage-2 server KMeans(M=32) │ ▼ codebook (M,64) + cluster-mean 잔차 → offsets (M,24) ┌──────────────────────────┐ test: │ ŷ + α·offset[1-NN(h_g)] │ α=1.0 └──────────────────────────┘ ⚠ backbone forward는 codebook을 절대 안 거침 (no commit, no gradient — 순수 분석)
  • codebook은 backbone을 관찰만 한다. 표현을 바꾸지 않는다.
  • 매 라운드 처음부터 다시 fit → collapse 구조적으로 불가능.
  • 5개 backbone에 동일하게 얹어 직교성(universality) 검증.

V9-FedVQ — 학습형 VQ 메모리

[FedAvg backbone] NBEATSxAuxVQ │ ▼ h_g ──► VQ codebook (EMA) │ │ argmin‖h_g−c‖² │ ▼ │ q_h_g (straight-through) │ │ ▼ ▼ commit = β·‖h_g − sg(q)‖² β=0.25 │ ▼ L = MAE + commit + λ·aux → backward │ (co-train!) ▼ server: mass-weighted + EMA blend(γ=.95) + dead-code respawn(매 K round) ⚠ codebook이 h_g를 형성한다 offset 보정은 별도 단계(plan §4), 02엔 없음
  • codebook과 backbone이 같은 gradient로 co-train된다.
  • collapse를 막기 위한 3중 장치(EMA reset·respawn·fedkmeans init) 필요.
  • 현재 FedAvg backbone 전용 — 5종 확장은 §4 ladder의 설계 결정.

3. 세 직교 메커니즘으로 분해 (R / A / C)

두 driver의 차이는 결국 세 축의 on/off로 환원된다:

축의미 RoundCBFedVQ
Rcluster 구조가 학습 중 loss로 backbone에 되먹임되는가 ✗ 없음 — hg는 codebook을 모름 ✓ commitment loss = FedProto proto-MSE의 VQ판
Aclient 간 cluster를 어떻게 집계하나 2-stage KMeans (count-weighted Stage-2) mass(count)-weighted + EMA blend
Ctest에서 cluster-mean 잔차 offset을 더하나 ✓ ŷ+α·offset (출력에 영향 주는 유일 경로) △ plan엔 있으나 02 미구현
비대칭의 본질: RoundCB는 출력은 보정하지만 표현은 안 건드리고(A+C), FedVQ는 표현은 형성하지만 출력 보정은 (아직) 안 한다(R+A). 즉 두 driver는 같은 codebook을 서로 반대쪽 끝에서 쓰고 있다. 둘을 합치면 "표현 형성 + 출력 보정을 같은 학습형 codebook으로 통합"이 되는데, 그것이 v09가 노리는 종착점(아래 B4)이다.

4. FedProto의 가설적 포지션 — co-train의 선례·ablation

FedProto는 prototype을 λ·MSE(h_g, proto[c])로 backbone에 되먹이고(=R축) count-weighted로 집계한다(=A축). FedVQ는 이 둘을 그대로 일반화한 것이다: (i) KMeans centroid+MSE → 학습형 VQ codebook, (ii) representation-only → 표현+출력 보정 통합. 따라서 FedProto는 "또 하나의 backbone"이 아니라 co-train clustering의 published 선례(AAAI 2022)이자 representation-only ablation으로 앉는다.

왜 RoundCB와 FedVQ가 둘 다 필요한가: v09의 진짜 주장은 단일 셀이 아니라 B4 vs B1 비교다. RoundCB(B1)가 "같은 codebook을 사후로 썼을 때의 lift"를 측정하는 통제군이고, FedVQ(B4)가 "co-train으로 통합했을 때의 lift"를 측정한다. FedProto(B2·B3)는 그 사이에서 "표현은 co-train했지만 통합은 안 한" 중간 칸으로, B4가 이기면 이득의 출처를 "표현+보정을 하나의 학습형 codebook으로 통합한 것"이라고 귀속시켜 준다.

5. 구현 현황 vs 계획 (정직 박스)

보고서를 실험에 그대로 쓰기 전 반드시 인지할 두 gap:
  • FedVQ는 현재 FedAvg backbone 전용. plan G2/Phase 1은 5종 전부(B-FedAvg…B-FedProto)를 적었지만 02는 weighted_average 하드코딩이라 알고리즘 스위치가 없다. B5(FedProto+FedVQ)는 코드 확장 필요.
  • FedVQ는 offset 보정(C축)을 아직 안 쓴다. 02의 _eval_per_client는 raw ŷ만 평가한다. plan §4의 08_apply_correction.py(미구현)가 offset을 붙이는 별도 단계다. 따라서 현 시점 B4의 lift는 순수 co-train(R) 효과이지 "통합 보정"의 완전체가 아니다.

6. v06 핵심 가설과의 정합성

미세한 긴장이 하나 있다. v06의 중심 주장은 "codebook은 method-agnostic·orthogonal 보정 모듈"(어떤 backbone에든 균등 lift)이다. FedProto를 동족 ancestor로 격상하고 co-train을 메인으로 삼으면 "codebook이 FedProto식 clustering과 특별한 관계가 있다"는 뉘앙스가 생긴다.

해소는 두 층위 분리로 한다:

  • v06 층위(post-hoc·보존): "보정 모듈은 어떤 backbone에든 재사용 가능한 직교 lift" — RoundCB가 per-round로 재현(B1, universality).
  • v09 층위(co-train·신규): "그 보정을 co-train하면(FedProto가 선례를 보인 방식) post-hoc 대비 추가 이득" — FedVQ가 검증(B4>B1).

이렇게 하면 v06 universality를 부정하지 않고 그 위에 v09를 쌓는다. FedProto는 두 층위를 잇는 다리가 된다.

7. 권고 / 다음 단계

  1. 메인 결과는 B4(FedVQ on FedAvg)에서 isolation, B1(RoundCB)을 직접 통제군으로 나란히 보고.
  2. FedProto는 B2/B3 ladder 칸으로 배치 — novelty 방어 + R-only ablation.
  3. FedVQ에 offset 보정(C) 추가(08_apply_correction.py)해 B4를 "통합 보정" 완전체로 만든 뒤 B1과 비교해야 가설이 완결된다.
  4. B5(FedProto+FedVQ)는 page budget 되면 redundancy ceiling ablation으로만.
  5. 논문 키워드 후보: Residual Prototype Memory · Prototype-guided Residual Forecasting · Federated Residual Codebook.
